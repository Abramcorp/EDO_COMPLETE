# ADR-004 — Data flow декларации КНД 1152017

**Статус:** Accepted
**Дата:** 2026-04-23
**Связанные:** ADR-001 (architecture), ADR-002 (pixel-perfect rendering), ADR-003 (receipts)

## Контекст

Декларация УСН (КНД 1152017) — главный документ, ради которого существует
pipeline. В отличие от квитанций (ADR-003), генерация тела декларации
сложнее: нужна бухгалтерская логика, данные от разных операторов ЭДО
(ТЕНЗОР, КОНТУР, возможно Контур.Эльба) имеют разную структуру хранения,
но итоговый PDF должен быть pixel-perfect бланком формы 1152017 (приказ
ФНС ЕД-7-3/1036@ и обновления).

На момент написания ADR в репо находятся:
- Три xlsx-шаблона (`declaration_template.xlsx`,
  `declaration_template_2024.xlsx`, `declaration_template_2025.xlsx`) — от
  родительского проекта usn-declaration. В них 0 формул — это чисто
  визуальные шаблоны формы.
- Модуль `modules/declaration_filler/` с tax_engine, revenue_calculator,
  contributions_calculator — то есть **вся бизнес-логика расчёта в Python**.
- `pdf_overlay_filler.py` — заглушка без реализации.

## Решение

### Итоговая архитектура (гибрид)

```
                  ┌──────────────────────┐
  ТЕНЗОР xlsx ──▶ │                      │
                  │  xls_reader_tensor   │
                  │  ──────────────────  │
  1C/ОФД   ──────▶│  parse_1c_statement  │
                  │  parse_ofd           │──┐
                  │  classify_operations │  │
                  └──────────────────────┘  │
                                            ▼
                              ┌──────────────────────────┐
                              │  tax_engine_calculate()  │
                              │  revenue_calculator      │
                              │  contributions_calculator│
                              └──────────────────────────┘
                                            │
                  ┌──────────────────────┐  ▼
  КОНТУР xlsx ──▶ │  xls_reader_kontur   │  ┌───────────────────┐
                  │                      │─▶│ DeclarationData   │ ← единый DTO
                  └──────────────────────┘  └───────────────────┘
                                                      │
                                                      ▼
                                  ┌────────────────────────────────┐
                                  │  pdf_overlay_filler.render()   │
                                  │  blank_YYYY.pdf + fields_YYYY.json│
                                  │  reportlab canvas + pypdf merge│
                                  └────────────────────────────────┘
                                                      │
                                                      ▼
                                               decl_body.pdf
                                                      │
                                                      ▼
                                       apply_stamps(operator=ТЕНЗОР|КОНТУР)
                                                      │
                                                      ▼
                                               final_decl.pdf
```

### Ключевые принципы

1. **XLS — источник или аудит, не калькулятор.** Бухрасчёт ведётся в Python
   (`tax_engine.py`). XLS-шаблоны используются двумя способами:
   - Как *входной формат* — когда ТЕНЗОР или КОНТУР выгружают уже
     рассчитанные значения в своём формате xlsx. Мы их *читаем*, не считаем.
   - Как *reference/аудит* — бухгалтер может открыть файл и сверить с
     тем, что увидит в Личном кабинете ФНС.

2. **Единый DTO `DeclarationData`.** Оба reader'а (tensor, kontur) и наш
   собственный расчётчик возвращают один и тот же dataclass. Рендер знает
   только о нём. Это изолирует формат источника от формата вывода.

3. **Один рендер, не два.** Тело декларации (стр. 1-4, форма КНД 1152017)
   одинаковое вне зависимости от оператора. Один `blank_YYYY.pdf`, один
   `fields_YYYY.json`, один код рендера. Разница между операторами — ТОЛЬКО
   в footer-штампе, который накладывается `apply_stamps` отдельным шагом.

4. **Версионирование по году.** Формы КНД 1152017 менялись: 2023 (приказ
   ЕД-7-3/1036@), 2024, 2025. Мы поддерживаем 2024 и 2025 — каждой свой
   `blank_YYYY.pdf` + `fields_YYYY.json`. Выбор версии — по `tax_period_year`
   в DTO.

5. **Операторо-специфичная разница видна только во 2-х местах:**
   - `xls_reader_*` — как распарсить входной xlsx
   - `apply_stamps(operator=...)` — какой штамп положить в footer

### Отличия от квитанций (ADR-003)

Квитанции проще: один жанр документа, одна структура, Footer-штамп один на
страницу. В декларации:
- 4 страницы тела с навигацией между разделами (Р.1.1, Р.1.2, Р.2.1.1, ...)
- ~100 полей данных, некоторые многострочные (например, адрес)
- Знакоместные клетки (char_cells) для ИНН, ОКТМО, сумм — каждая цифра в свою клетку
- Checkbox-поля (объект налогообложения, форма реорганизации)
- Страничные номера 001/002/003/004

Поэтому `fields_YYYY.json` для декларации сложнее, чем для квитанций, и
разметка — многократно больший труд.

## Последствия

### Положительные

- Чёткая граница между источниками данных и рендером
- Переключение операторов = замена одного класса reader, остальное не трогаем
- Тестируемость: `DeclarationData` легко сконструировать в тесте без xlsx
- Отсутствие LibreOffice в runtime зависимостях (xlsx читается openpyxl'ом
  только для значений, не для рендера)

### Компромиссы

- Требуется ручная разметка `fields_2024.json` и `fields_2025.json` (объём работы)
- XLS-шаблоны в репо сейчас не имеют формул — не являются калькуляторами.
  Если понадобится ручная работа бухгалтера в xlsx (ввёл данные — получил
  суммы), нужен будет отдельный PR по добавлению формул. Для MVP достаточно
  Python-расчёта.
